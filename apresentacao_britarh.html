03

Domínio & Clean Arch

O coração do sistema é o Domínio, independente de banco de dados ou APIs.

  • Model / Entidade: Representa a verdade do banco (SQLAlchemy).
  • Schema: Define o contrato da API e validação (Pydantic).
  • Regra de Negócio: "Vagas só aceitam currículos se estiverem ativas".

Model = Persistência
Schema = Contrato/Validação
Service = Domínio/Orquestração

Essa tríade garante que regras de negócio não se misturam com detalhes técnicos de infraestrutura.

05

A Entidade: O Coração do BritaRH

No centro do sistema está a entidade Job (Vaga). No DDD, uma entidade é definida por sua Identidade, não apenas seus atributos.

  • Identidade Única: UUID persistente.
  • Estado: Transição entre OPEN e CLOSED.
  • Invariantes: Uma vaga só é válida se possuir no mínimo 5 questões técnicas.
class Job(Base):
  __tablename__ = "jobs"
  id: Mapped[UUID] = mapped_column(primary_key=True)
  title: Mapped[str] = mapped_column(index=True)
  status: Mapped[JobStatus] = mapped_column(default=JobStatus.OPEN)
  # Relacionamento Forte (Aggregate Root)
  questions: Mapped[list["Question"]] = relationship(...)
